# 执行评估准则

当 TODO 文档措辞含糊、完成标准容易被弱化，或你需要判断某项是否能诚实标记为完成时，使用这份准则。

## 目录

- 保持顺序
- 严格定义完成
- 立即同步 TODO
- 识别真实阻塞
- 避免作弊
- 最大化真实吞吐

## 保持顺序

- 将文档顺序视为默认优先级。
- 除非文档明确说明可并行或可重排，否则不要自行调整先后顺序。
- 标题通常代表阶段；不要跨阶段挑更容易的任务先做。
- 当前任务只要还能推进，就先完成当前任务。

## 严格定义完成

只有当条目文本直接要求的交付物已经存在，而且最相关验证已经通过时，任务才算完成。

示例：

- 代码任务：行为已实现；相关调用点已同步；最相关测试或构建已通过。
- 修 bug 任务：问题已复现或通过上下文确认；修复已落地；验证证明错误不再出现。
- 重构任务：结构已改善；外部行为保持正确；回归验证通过。
- 文档任务：目标文档已更新；内容与实际实现一致；不是只写临时笔记。
- 调研任务：最终结论、对比或建议已经产出；不是只停留在原始搜索记录。

## 立即同步 TODO

- 每完成并验证一个条目，就立刻更新 TODO 文档，不要积压到最后统一勾选。
- 只有完成了真实工作，才允许勾选复选框。
- 文档若没有复选框，不要强行改格式；在汇报中说明哪些条目已完成即可。
- 如果一个条目完成后暴露出与之绑定的小型补充工作，应先补完，再勾选。

## 识别真实阻塞

以下通常属于真实阻塞：

- 缺失权限、凭证、网络访问、硬件、外部服务或必需的第三方系统。
- 与当前任务无关但会污染结论的基线失败。
- 文档或仓库指令互相冲突，且不同解释会产生不同用户可见结果。
- 缺少用户必须做出的产品/架构选择，继续执行会导致高概率返工。

以下通常不属于真实阻塞：

- 任务麻烦、耗时、文件多、需要多步实现。
- 需要先阅读更多上下文。
- 需要自己补一两个小测试、脚本或辅助改动。
- 后面还有更容易完成的条目。

## 避免作弊

- 不要把分析、草稿、占位符、`TODO`、`FIXME` 或空壳函数说成完成。
- 不要通过修改任务描述、删除未完成项或模糊措辞来制造完成假象。
- 不要省略明明应该做的验证，也不要虚报命令结果。
- 不要把“已经做了一部分”说成“已经完成”。
- 不要因为赶进度而跳过失败验证。

## 最大化真实吞吐

- 完成一个条目后，趁上下文还热，直接进入下一个未完成条目。
- 若相邻条目共享上下文，可连续推进，但仍要逐项验证、逐项回写 TODO。
- 若一个条目自然展开出必要的隐含小步骤，立刻做完，不要留下隐藏债务。
- 只在真实阻塞、风险决策缺失或用户明确打断时停止。
